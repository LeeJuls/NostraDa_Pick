# 노스트라 다 찍기 - 스케줄 제어 및 수동 관리 히스토리 (v1.0)
*작성일: 2026-03-01*

## 1. 개요 및 논의 사항
- 무료 Gemini API의 할당량 한계로 인해, 자동 스케줄러(1분 주기) 및 고성능 모델 사용 시 빠른 429 에러(Quota Exceeded)가 발생하여 테스트에 큰 차질이 발생했습니다.
- 개발(로컬) 환경에서 손쉽게 문제를 출제하고, 결과를 임의로 정산할 수 있는 '수동 관리(Admin) 기능'의 필요성이 제기되었습니다.
- 테스트용 로컬 DB 운영에서 실 서비스와 동일한 Supabase 프로젝트를 사용할 때 발생하는 데이터 혼선(랭킹 등) 문제 논의가 있었습니다.

## 2. 주요 변경 내역
### 2.1. API & 모델 다중화 및 폴백(Fallback) 기능
- `config.py`, `services/gemini_service.py` 수정:
  - `GEMINI_API_KEYS`를 다중 키로 활용하여 Quota 초과 시 교체(Swap)되도록 로직 개선.
  - 최신 2.0-flash-lite 모델 사용.
  - 모든 API 키가 소진되었을 때 시스템이 뻗지 않고, 테스트(로컬) 환경이라면 더미 이슈 모음에서 랜덤 추출하여 생성하는 `_generate_fallback_issues()` 폴백 기능 추가.

### 2.2. 로컬 관리자(수동) 기능 - 문제 출제, 결과 정산
- `routes/api.py`, `static/js/app.js` 추가:
  - `/api/admin/force-issue-gen`: 로컬 환경에서 Gemini API를 한 번 강제 호출하여 문제를 실시간으로 출제(`reset_and_generate` 로직 재활용).
  - `/api/admin/force-resolve`: 현재 OPEN 상태인 이슈들에 무작위 정답(correct_option_id)을 배정하고 CLOSED 처리하여 정산. 이때 관련된 사용자들에게 점수 증감(지급/삭감) 로직 추가 (`routes/api.py` 내부 및 랭킹 조회 `get_my_stats()`에서의 정답 컬럼명 불일치 버그 수정 포함).
  - 프론트엔드(`app.js`)에서 `localhost` 접속 시에만 숨겨진 Admin Panel이 나타나도록 예외 처리.

### 2.3. DB 분리 원칙 설정 (문서화 반영)
- 라이브 배포 시에는 스케줄러를 `cron` (UTC 0, 4, 8, 12, 16, 20시) 기반으로 복귀하였고, 로컬 테스트는 별도의 프로젝트, 혹은 더미 로직으로 돌아가게 격리시킴.

## 3. 결과 및 검증 과정
1. **문제 자동 생성 검증**: 백그라운드 환경 외에 사용자가 `강제 생성` API를 트리거링할 시 이슈가 즉시 3건 자동 생성됨을 확인. (단, API Limit 시 로컬에서는 더미 데이터로 동작함 검증)
2. **랜덤 정산 검증**: 3건의 이슈가 OPEN 되어있는 상황에서 `강제 정산` API 실행 검증 완료 (모든 이슈 status가 RESOLVED로 변경되고, `correct_option_id`에 옵션 id가 제대로 삽입됨을 파이썬 스크립트로 확인).
3. **버그 수정 (컬럼명 오류)**: `_resolved_option_id`로 코딩되었던 스크립트들을 실제 DB의 `correct_option_id` 스키마 이름과 동일하게 모두 통일 수정 완료함.
