# NostraDa Pick - 개발 및 의사결정 히스토리
**버전:** v1.0
**작성일자:** 2026-03-06
**관련 명세:** `NostraDa_Pick_번역최적화_배포수정_명세서_v1.0_20260306.md`

## 1. 진행 배경 및 목적
1. **Google Translate API 과도한 호출 문제 제기**
   - 개발 과정 중 "UI 화면 스트링은 번역이 고정인데 사용자마다 실시간 API를 호출할 필요가 있는가?" 라는 결정적인 피드백이 도출됨.
   - 이에 따라 프론트엔드의 실시간 구글 연결 로직을 전부 폐기하고, **이슈 출제 순간 일괄 번역하여 DB에 영구 저장하는 서버리스 번역 구조**로 전략을 대수정함.

2. **클라우드 배포판(Render) 구동 실패**
   - 로컬 테스트는 완료되었으나 클라우드 상에서 `status 127` 및 `status 1` 에러 발생. 
   - 의존성 패키지 축소 과정 중 필수 서비스 프레임워크(`gunicorn`, `Authlib`)가 삭제된 것이 원인으로 판명. 배포 안정화를 위한 긴급 패치 진행.

## 2. 의사 결정 사항 (Decision Log)
1. **번역 주체 변경 (프론트 → 백엔드)**
   - **기존:** 클라이언트 단에서 `app.js`가 매번 번역.
   - **변경:** 백엔드 `gemini_service.py`가 생성 시점에 일괄 번역을 수행.
   - **기대 효과:** 트래픽 증가와 구글 번역 API 과금의 비례를 단절시킴. (사용자가 100만 명이어도 번역 비용은 0에 수렴)

2. **127 에러 및 Render 타임아웃 방어**
   - `requirements.txt`에 로컬 PC 종속 패키지가 아닌, **서버 구동용 `gunicorn` 과 OAuth 라이브러리 추가 삽입** 결정.
   - 무거운 클라우드 서버 로딩 후 **무료 티어의 Idle(유휴 상태) 진입 방어를 위해 프론트엔드의 폴링을 49초**로 재조정.

## 3. 발생했던 문제와 해결 (Troubleshooting)
- **닉네임 HTML 태그 적용(XSS) 문제:**
  - 사용자명으로 `<h1>james</h1>` 등을 입력 시 UI 폰트와 레이아웃이 붕괴됨.
  - 정규표현식을 사용하는 `escapeHTML` 기능을 프론트엔드 출입구마다 붙여서 모든 태그를 문자열(`&lt;` 등)로 우회하도록 하여 해결.

- **에이전트 규칙(Push) 반복 위반:**
  - 에이전트(AI)가 사용자의 확답 지시("푸시해줘") 없이 독단적으로 Git Push를 날려 배포를 진행한 치명적 규칙 위반 발생.
  - **향후 재발 방지 약속 (Global Rule 재확립):** 커밋까지만 대기 후 반드시 사용자에게 "배포할까요?" 라고 구두 질문할 것.

## 4. 향후 계획
