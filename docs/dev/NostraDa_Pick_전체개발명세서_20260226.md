# NostraDa_Pick: 전체 개발 스케줄 및 확장 명세서 (v1.0)
작성일자: 2026-02-26

## 1. 프로젝트 개요
**NostraDa_Pick**은 AI(Gemini)가 생성한 경제, 사회, 스포츠 등의 이슈에 대해 사용자가 'Yes/No' 중 하나를 선택하여 포인트를 베팅하는 **예측 시뮬레이션 게임**입니다.

### 💡 [PM/Front/Back 논의] 핵심 재미 및 리텐션(잔존율) 강화 방안
단순한 Yes/No 투표를 넘어 사용자 재방문율(Retention)을 높이기 위해 팀 회의를 통해 다음과 같은 요소를 추가 반영하기로 결정했습니다.

1. **🔥 실시간 배당률 (Dynamic Odds):** 양쪽 선택지에 걸린 포인트 비율(pool_amount)에 따라 예상 수익률(배당률)을 실시간으로 화면에 보여주어 눈치싸움의 재미를 극대화합니다.
2. **📈 레벨/랭킹 시스템 도입:** 사용자의 '누적 적중 횟수' 또는 '보유 포인트'를 기반으로 랭킹 보드를 띄우거나, 레벨 뱃지(예: 풋내기 예언가 -> 노스트라다무스)를 부여합니다.
3. **💬 미니 댓글/응원 시스템 (선택):** 각 문제 하단에 가벼운 응원이나 의견을 남길 수 있도록 하여 커뮤니티성을 한 스푼 더합니다. (2단계 확장 목표)

## 2. 전체 개발 스케줄 (Milestones)

### 🗓️ Phase 1 (Week 1): 코어 백엔드 및 인프라 구축
- **목표:** DB 스키마 완비 및 Supabase 통신 확립
- [Back] `.env` 환경 변수 최종 점검 및 `supabase_client.py` 연결
- [Back] `users`, `issues`, `options`, `bets` 4대 핵심 테이블 Supabase 생성 스크립트 작성
- [Back] Flask에서 DB 데이터를 CRUD하는 기본 API 라우팅 (`routes/api.py`) 구성
- [QA] Supabase 연결 안정성 및 DB 입력/조회 에러 테스트

### 🗓️ Phase 2 (Week 1~2): 제미나이(AI) 문제 출제 자동화 파이프라인
- **목표:** 사람이 개입하지 않아도 AI가 트렌디한 문제를 자동으로 출제
- [Back] Gemini API 호출을 위한 프롬프트 엔지니어링 (JSON 형태로 반환받기)
- [Back] 받아온 문제(Title, Category, Close_time)를 `issues`, `options` 테이블에 Insert 하는 엔진 스크립트 작성
- [PM] 운영체제 스케줄러(Cron) 또는 1회성 배치 스크립트 구동 방식 결정
- [QA] Gemini Rate Limit 도달 시 백업 로직(과거 문제 재활용 등) 작동 여부 검증
- [QA] AI가 생성한 JSON 형식이 깨졌을 때의 파싱 에러 방어 로직 검증

### 🗓️ Phase 3 (Week 2): 프론트엔드 UI/UX 구현 (리텐션 요소 반영)
- **목표:** 사용자 친화적이고 직관적인 웹 화면 구성
- [Front] 메인 대시보드 구조화 (네비게이션 바, 랭킹 보드 영역, 이슈 카드 리스트)
- [Front] 이슈 카드 UI 디자인 (마감 타이머 구현, 'Yes/No' 버튼 및 **실시간 배당률 게이지바** 표시)
- [Front] Flask Jinja2 템플릿과 연동하여 DB의 실제 데이터를 화면에 렌더링
- [QA] 모바일/데스크탑 반응형 브라우저 렌더링 검수
- [QA] 버튼 연속 클릭(다중 베팅 시도) 방어를 위한 디바운싱/스로틀링 로직 체크

### 🗓️ Phase 4 (Week 3): 베팅 트랜잭션 및 정산 로직 고도화
- **목표:** 돈(포인트)이 오가는 핵심 로직의 무결성 확보
- [Back] 베팅 시 사용자의 포인트 차감 및 `bets`, `options(pool_amount)` 업데이트를 **동시(Transaction)** 에 처리하는 Supabase RPC(Stored Procedure) 작성 (**매우 중요**)
- [Back] 마감 시간이 지난 문제에 대해 결과를 판정하고 수익금을 분배하는 정산(Resolve) 스크립트 작성
- [QA] **[Edge Case]** 베팅 마감 1초 전 수십 명이 동시에 요청할 때의 DB Lock/Race Condition 현상 부하 테스트
- [QA] **[Edge Case]** 한쪽 옵션에만 100% 몰려서 반대쪽 배당금이 0이 되는 상황(0으로 나누기 에러 등) 방어 점검

### 🗓️ Phase 5 (Week 3~4): 최종 폴리싱 및 릴리즈 준비
- **목표:** 실서비스 수준의 안정성 확보 및 이력 정리
- [PM] 개발 완료 항목 목록화 및 `docs/history/` 문서 최종 산출
- [QA] 전체 유저 플로우(회원가입 -> 베팅 -> 마감 -> 정산 -> 랭킹 변동) E2E 테스트 
- 전 파트 버그 픽스 후 GitHub main 브랜치 안정화 작업 (완료 시 커밋/푸시 대기)
