# NostraDa_Pick: QA 테스트 시나리오 및 예외처리 가이드 (v1.0)
작성일자: 2026-02-26

## 1. 개요
NostraDa_Pick (노스트라 다 찍기) 시스템의 퀄리티 보증(QA)을 위한 단계별 테스트 시나리오 문서입니다. 모든 개발 파트는 기능 구현 후 아래 항목을 1차 자가 테스트(Unit/Local) 해야 하며, 통합 시 QA 담당자가 최종 검증합니다.

## 2. 모듈별 테스트 시나리오 (Test Cases)

### 🧪 TC-1: 백엔드 & DB 인프라 (Phase 1)
- **TC-1.1 [DB 연결]:** `.env` 설정값을 변경/삭제했을 때 서버 기동 시 명확한 에러 메시지(Fallback)가 출력되는가?
- **TC-1.2 [CRUD 무결성]:** `users` 테이블에 중복된 이메일로 가입 시도 시 DB단에서 UNIQUE 제약조건으로 반려되는가?
- **TC-1.3 [참조 무결성]:** `issues` 레코드를 삭제(CASCADE)할 때, 하위 `options`와 `bets` 테이블의 관련 데이터도 정상적으로 정리/보존(설계 정책에 따라) 처리되는가?

### 🧪 TC-2: 제미나이 AI 스케줄러 (Phase 2)
- **TC-2.1 [API Rate Limit]:** 단기간 내에 할당량 이상의 출제 요청을 보냈을 때 (429 Too Many Requests), 프로세스가 죽지 않고 재시도 대기(Exponential Backoff) 로직이 작동하는가?
- **TC-2.2 [응답 파싱 오류]:** Gemini가 약속된 JSON 포맷(Title, Category 등)이 아닌 일반 텍스트 등을 반환했을 때, `try-except`로 걸러내고 패스(Pass)하는가?
- **TC-2.3 [시간 논리 오류]:** 출제된 문제의 `close_at` (마감시간)이 현재 시간보다 과거로 설정되어 반환되었을 경우 거부하는가?

### 🧪 TC-3: 프론트엔드 UI/UX (Phase 3)
- **TC-3.1 [다중 클릭 방지]:** 투표 버튼 연속 클릭 시, 첫 번째 요청만 서버로 전송되고 버튼은 로딩/비활성화(Disabled) 상태로 즉시 변경되는가? (디바운싱 확인)
- **TC-3.2 [렌더링 에러]:** 서버에서 긴 제목의 이슈(글자 수 제한 초과 등)가 내려올 때 카드 UI의 CSS 레이아웃이 깨지지 않는가?
- **TC-3.3 [게이지 바 애니메이션]:** 한 쪽 옵션에만 100% 자본이 몰려 있을 때, 배당률/투표율 바 UI가 에러 없이 0%~100% 비율을 정상 렌더링하는가?

### 🧪 TC-4: 트랜잭션 및 정산 코어 로직 (Phase 4) - 🚨 제일 중요
- **TC-4.1 [동시성 부하 테스트]:** 베팅 마감 1초 전, 유저 100명이 동시에 베팅(Insert)을 시도할 때 `options.pool_amount` 값이 분실(Race Condition) 없이 100건 모두 정확히 합산되는가?
- **TC-4.2 [잔액 부족 방어]:** 유저가 보유한 포인트(Point)보다 더 많은 액수를 베팅하려 할 때 백엔드(RPC)에서 안전하게 차단하는가?
- **TC-4.3 [마이크로 초 차이 마감]:** 마감 시간 정각(`close_at`) 이후에 도착한 늦은 베팅 요청은 정확하게 반려(Rejected) 처리되는가?
- **TC-4.4 [Zero Divide 예외]:** 결과가 `A` 옵션으로 확정({`correct_option_id`})되었으나, `A`에 건 사람이 아무도 없어서 배당금을 나눌 때 분모가 0이 되는 상황의 예외(수익금 회수 등 정책) 처리가 되어 있는가?
