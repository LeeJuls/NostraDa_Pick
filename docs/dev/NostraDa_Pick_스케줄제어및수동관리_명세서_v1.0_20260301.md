# 노스트라 다 찍기 - 스케줄 제어 및 수동 관리 명세서 (v1.0)
*작성일: 2026-03-01*

## 1. 개요 및 문제 시나리오
- **문제 1**: 무료 Gemini API의 할당량 한계로 인해, 자동 스케줄러(1분 주기) 및 고성능 모델 사용 시 빠른 429 에러(Quota Exceeded)가 발생함.
- **문제 2**: 개발(로컬) 환경에서 손쉽게 문제를 출제하고, 결과를 정산(Yes/No 임의 배정)할 수 있는 '수동 관리(Admin) 버튼'이 없음.
- **문제 3**: 개발(로컬) 환경과 실제(라이브) 배포 서버가 동일한 DB 테이블(특히 `points` 및 `users` 랭킹)을 사용하게 되어 데이터 혼선이 우려됨.

## 2. 해결 및 변경 계획 (개발 내용)

### [수정 1] 테스트용 다중 API & 하위 모델 로직 적용
- **대상 파일**: `config.py`, `services/gemini_service.py`
- **변경 사항**: 
  - `GEMINI_API_KEYS` 쉼표(,) 구분자를 통한 다중 키 셋업.
  - 일일 1500건 수준의 할당량을 제공하는 `gemini-1.5-flash-latest` 대신 사용 가능한 `gemini-2.0-flash-lite` 또는 `gemini-1.5-flash` 모델 세팅 및 `429` 에러 발생 시 키 자동 교체(Swap) 로직.

### [수정 2] 관리자(수동) 기능 - 수동 문제 출제 및 결과 처리
- **대상 파일**: `routes/api.py`, `app.py`, `templates/index.html`
- **변경 사항**:
  - `admin/force-issue-gen` API: Gemini API를 한 번 강제 호출하여 문제를 출제 (`reset_and_generate` 로직 활용)
  - `admin/force-resolve` API: 현재 활성화(OPEN)된 문제를 더미 결과(랜덤한 쪽이 이기게 하거나 요청 시 특정 옵션 선택)로 강제 종료하고 정산 처리하는 API (개발 기간 테스트용).

### [수정 3] 로컬과 라이브 서비스의 DB 분리 지침
- **대상 사항**: `app.py` 환경변수 또는 `.env` 설정
- **변경 방향**:
  - 동일한 Supabase 프로젝트를 사용하되, 테이블(예: `dev_issues`, `dev_users`)을 분리하거나, 로컬에서만 사용하는 별도의 Supabase 프로젝트(URL, KEY)를 `.env`에 등록하여 완전 격리하는 방안 적용.

## 3. 테스트 및 검증(QA) 계획
1. 프론트엔드에 추가할 'Admin 수동 생성 버튼'을 누르면 즉시 새로운 이슈가 로컬 DB에 뜨는지 확인.
2. 'Admin 결과 생성 버튼'을 누르면 이슈가 닫히면서 승/패가 나뉘고 랭크에 반영되는지 확인.
3. 이 모든 과정이 로컬 환경 전용 DB(또는 별도 프로젝트)에서만 돌아가는지 상호 크로스체크.
