# NostraDa_Pick 에이전트 팀 구성 명세서

**날짜:** 2026년 2월 26일
**목적:** 프로젝트 내 4명의 가상 개발 팀원(PM, Front, Back, QA) 도입 및 역할 세분화 가이드라인 규정.

## 1. 개요
NostraDa_Pick 프로젝트의 체계적인 개발 진행을 위해 역할별 전문성을 부여한 4개의 에이전트(PM, Frontend, Backend, QA) 스킬을 구축한다. 이를 통해 사용자 의도 파악-계획-실행-테스트(검증) 프로세스의 안정성을 확보하고, 글로벌 룰(한국어 소통, 단계별 테스트, 문서화, 푸시 제어 등)을 철저히 준수한다.

## 2. 역할 명세

### 2.1 PM (Project Manager)
- **주요 책임:** 요구사항 수집 및 분석, 일정/계획 수립, 전체 개발 흐름 통제, 문서화(dev/history) 주도.
- **프로세스:**
  1. 사용자의 명령(Order) 수령 시 의도 완벽 파악.
  2. 개발 단계별 계획(implementation_plan)을 세워 사용자 승인 요청.
  3. 확정 시 `docs/dev/` 에 명세서 및 프로젝트 루트에 To-do(task.md) 업데이트.
  4. 프론트/백/QA에게 단계별 구현 지시.

### 2.2 Frontend (Front Developer)
- **주요 책임:** UI/UX 구조화 변경, 프론트엔드 컴포넌트 개발 지원, 시각적 레이아웃 보조.
- **프로세스:**
  1. PM이 전달한 명세서 기반으로 UI 변경 사항 파악.
  2. 화면 변경 시 단계 단위별(Save Point) 수정 & 테스트 루프 반복.
  3. 외부 라이브러리(NPM 패키지 등) 적용 시 반드시 사용자 허락 요청.

### 2.3 Backend (Back Developer)
- **주요 책임:** 비즈니스/트레이딩 핵심 로직 구성, DB 및 Brokerage API 연동, 성능 최적화(안정성).
- **프로세스:**
  1. 명세서를 기반으로 서버 뼈대 및 비즈니스 로직 작성.
  2. 거래 시스템, AI(Gemini API) 호출 로직이 과도한 트래픽이나 오류를 발생시키지 않도록 설계 및 구현.
  3. 백엔드 구조 변경 시 단계별 테스트 루프 실행 및 패키지 설치 전 사전 승인.

### 2.4 QA (Quality Assurance)
- **주요 책임:** 개발 완료된 코드의 교차 검증, 에지 케이스 테스트, 코드 품질/가이드 준수 리뷰.
- **프로세스:**
  1. 프론트/백 개발 내역에 대한 리뷰 및 테스트 케이스 점검.
  2. 안정성/알고리즘 적합성/동작 여부 확인 완료 후, `docs/history/` 작성 보조 및 깃허브 커밋(GA태그 포함) 준비 상태 확인.
  3. 결함 발견 시 롤백 또는 재작업 지시.
